# The IR model

The intermediate representation (IR) is the heart of Titanbase. It is a JSON document (`.titan.json`) that describes a relational database schema independently of any specific engine.

A schema is made of a handful of building blocks:

- **Tables** — named collections of columns, with optional indexes and check constraints.
- **Columns** — typed fields with constraints like nullability, uniqueness, and defaults.
- **Relations** — foreign-key relationships between tables, with cardinality and referential actions.
- **Enums** — shared enumerated types reusable across columns.
- **Target hints** — optional, engine-specific overrides.
- **Metadata** — editor-only data such as canvas positions.

## Root object

```typescript
interface TitanSchema {
  $schema: string;        // JSON Schema URL for validation
  version: string;        // IR format version (semver)
  name: string;           // project identifier (kebab-case)
  description?: string;

  tables: Table[];
  relations: Relation[];
  enums?: EnumDef[];

  targetHints?: Record<string, unknown>;
  metadata?: {
    editor?: EditorMetadata;
    createdAt?: string;
    updatedAt?: string;
  };
}
```

## Model vs. output

The IR is **target-agnostic**: it never contains engine-specific syntax. A column is `{ "type": "uuid" }`, not `UUID` or `CHAR(36)`. Exporters are responsible for mapping abstract types to concrete ones for each target — see the [type system](/format/types).

This is what lets one `.titan.json` produce PostgreSQL, MySQL, SQLite, Prisma, and Drizzle output from a single definition.

## Determinism

The IR is serialized deterministically — keys are sorted alphabetically at every level. The same logical schema always produces byte-identical JSON, which keeps git diffs clean and meaningful. Editor-only state is isolated in `metadata` so layout changes never pollute schema diffs.

## Validation

`@titanbase/core` validates a schema before any export. A valid `.titan.json` must satisfy rules such as unique table names, unique column names within a table, at least one column per table, and relation references that point to existing tables and columns. See the full list in the [format reference](/format#validation-rules).

Continue to the [`.titan.json` format reference](/format) for field-by-field details.
